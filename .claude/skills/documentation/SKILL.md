# ドキュメント配置ルール

## 大原則

**事実は1箇所に書き、リンクで繋ぐ。実装が始まればコードが SSOT。**

## 各層の責務

| 層 | 書くもの | 書かないもの | 制約 |
|---|---|---|---|
| CLAUDE.md | 概要、スタック、構成、コマンド | なぜ、実装パターン、ルール | ~50行。@import 禁止(symlink互換) |
| rules/ | 命令だけ(「やれ」「するな」) | 理由 → decisions/ へリンク | 各 ~200トークン。paths で絞る |
| skills/ | 実装パターン + コード例、ワークフロー手順 | 制約の重複 → rules/ が権威 | description 1行。本文 < 5,000トークン |
| decisions/ | 決定 + 理由 + 影響(個別ファイル) | 実装手順、コード例 | status: active/superseded。インデックスは別途管理 |
| README.md | 「何のサイトか」+ 構成サマリー | 技術詳細、コマンド、設計方針 | 人間向け。AI 自動ロード外 |
| auto memory | AI が発見したパターン・学習 | 人間の設計意図 | プロジェクト正式知識とは分離 |

## SSOT 判定フロー

```
「この情報はどこに書く？」

AI に守らせたい制約？
  YES → rules/(paths で絞る)
  NO ↓

AI に実装パターンを教えたい？
  YES → skills/
  NO ↓

なぜその判断をしたか？
  YES → decisions/(個別ファイル)
  NO ↓

人間が読むプロジェクト紹介？
  YES → README.md
```

## 禁止パターン

- 同じ事実を2箇所に書く — 片方が腐る
- rules/ に「なぜ」を書く — 理由は decisions/ の個別ファイルへリンク
- CLAUDE.md を太らせる — 常時ロード = 毎セッションのコスト
- コードで表現できることをドキュメントに書く — スキーマはコード、型はコード
- 空ディレクトリを先に作る — 中身ができてから

## 新規ドキュメント作成時のチェックリスト

1. 既存のどこかに同じ情報がないか確認
2. 上の判定フローで配置先を決定
3. 他の層から参照が必要なら「リンク」で繋ぐ(コピーしない)
4. 制約の上限(トークン数、個数)を超えていないか確認

## このスキル自体の更新

運用で判断に迷ったケースや新しいパターンが出たら、このファイルに追記する。ドキュメント構造は継続的に育てるもの。
